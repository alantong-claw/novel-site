# Research Orchestration Rule

適用情境：使用者要求「研究」某主題時。

> 注意：這份文件是 `MULTI_AGENT_ORCHESTRATION.md` 在研究場景下的特化版本。
> 若發現辯論、提醒、分工任務也踩到相同問題，應優先修通用 orchestration 規則，而不是只補研究特例。

## 目標
避免研究流程卡在回合之間，讓主代理必須主動：
- 追蹤 round 狀態
- 在每一輪結果到齊後立刻續派下一輪
- 主動向使用者推送進度
- 在最後自動進入整理 / 輸出 / 寄送

## 固定流程

### 1. 啟動
- 先由主代理提出至少 3 個獨立研究面向（可更多）
- 每個面向都要獨立跑一套研究循環：
  - 至少建立兩個 subagent：
    - Subagent A：提出方案、架構、做法、證據、正向主張
    - Subagent B：提出反對意見、風險、缺口、反證與質疑
  - 預設至少 3 輪；若已明顯收斂，可提早停止
- 所有面向完成後，主代理再做跨面向 synthesis

### 2. 回合控制
每一輪都必須：
1. 記住本輪預期 completion event 的兩個 child session
2. 第一個結果到時：
   - 立刻用正常語氣對使用者推一則簡短進度摘要
   - 明講目前是第幾輪、哪一方先到、正在等另一方
3. 第二個結果到時：
   - 主代理不得停住
   - 必須在同一個工作鏈中立刻決定：
     - 是否發下一輪
     - 或是否已可進 final synthesis

### 3. Status push 規則
- 研究任務進行中，不能只在使用者追問時才更新
- 至少在以下時點主動推送：
  - 第 1 輪第一個結果到達
  - 每輪雙方都到齊並已續派下一輪
  - 已進 final synthesis / 正在產生文字檔、Excel、PPT
  - 已完成寄送
- Status 訊息要短，但要交代：
  - 現在在哪一輪
  - 是否已續派下一步
  - 下一個明確里程碑是什麼

### 4. 禁止停滯
以下情況視為流程錯誤：
- 第 N 輪雙方結果都到齊，但主代理沒有立刻發出第 N+1 輪
- 已答應會持續觀察，卻仍停在等待使用者追問
- 已完成最後一輪，但沒有立刻開始 synthesis / deliverables

### 5. 最終整理
最後必須由主代理完成：
1. 主代理自己的跨面向最後判斷
2. 完整文字報告（最好保留各面向、各回合的完整來回內容）
3. Excel 正反重點表
4. PowerPoint 報告版
5. 依使用者規則寄送郵件

### 6. 預設輸出節奏
推薦固定節奏：
- Round 1 first result → 推狀態
- Round 1 complete → 立即發 Round 2
- Round 2 first result → 推狀態
- Round 2 complete → 立即發 Round 3
- Round 3 first result → 推狀態
- Round 3 complete → 立即進 synthesis
- Synthesis started → 推狀態
- Mail sent → 推最終完成

## 核心原則
研究流程不是「子代理做完再看有沒有想到」，而是「主代理要像 PM 一樣主動推進」。
